# Auto-Transcribe 超完整工作流展示

## 这套工作流到底厉害在哪

这不是一个“把音频转成文字”的小工具，而是一条从**采集内容**、到**理解内容**、再到**沉淀为可复用知识资产**的全自动生产线。

它真正完成的是这件事：

**任何地方来的语音、视频、链接、录音，只要进入系统，就会被自动接住，自动识别，自动转录，自动总结，自动整理成结构化笔记，最后沉淀进 Obsidian。**

也就是说，你不是在“处理文件”，你是在搭一个属于自己的个人内容操作系统。

---

## 一句话总览

```text
多入口采集 → 统一进入 inbox → 本地转录 → AI 分类/命名/总结 → Obsidian 沉淀 → 周报再聚合
```

## 这条生产线的核心能力

### 1. 多入口统一收口

系统最强的第一层，不是转录，而是“接入能力”。

它不是只等你手动扔一个文件进来，而是已经把多种现实场景接成了一个统一入口：

- Telegram 发来一个链接，可以自动判断是视频还是文字内容
- Plaud 录音笔或 Plaud Desktop 录完，会自动从云端拉下来
- iPhone 上录的语音备忘录，分享到 iCloud 指定文件夹后自动同步
- 浏览器下载器可以直接把文件落到 `inbox/`
- 任意本地音频、视频也可以手动拖进去

这意味着你平时的采集动作不用改变，但所有内容最后都流进同一条处理管道。

### 2. 不是“存素材”，而是“自动加工素材”

大多数工具停留在“把文件保存下来”。

这套系统会继续往下做：

- 自动检测文件是否已下载完成
- 自动把视频拆成音频
- 自动本地 Whisper 转录
- 自动判断这段内容属于会议、备忘、课程、电话、讨论，还是外部视频内容
- 自动生成更像人写的标题
- 自动调用对应 prompt 生成结构化总结
- 自动写入 Obsidian，并附带 frontmatter、来源、时长、标签

也就是说，进入系统的不是“原材料仓库”，而是“自动加工厂”。

### 3. 从“内容流”变成“知识库”

处理完成后，产物不是一段散乱文本，而是可长期积累、可检索、可复盘、可再利用的知识笔记：

- 原始转录保留在 `transcripts/`
- 结构化总结写进 Obsidian
- 社交媒体文字抓取单独进入 `Captures/`
- 每周自动聚合成周报

最终效果是：你的语音输入、会议记录、外部内容消费，都会自动进入同一个知识系统。

## 完整工作流拆解

## 第一阶段：内容进入系统

### 入口 A：Telegram Bot 链接捕获

文件：`telegram-capture.py`

这是这套系统里最“像产品”的入口之一。你只需要把链接发给 Telegram Bot，它就会替你完成后面的判断和分流。

完整流程如下：

```text
用户发链接给 Bot
  ↓
解析可选语言前缀（en / zh / ja / ko / auto）
  ↓
展开短链接（t.co / bit.ly / xhslink / b23 / v.douyin 等）
  ↓
清理追踪参数（utm_* / fbclid / igshid 等）
  ↓
安全校验（限制协议，阻止私有 IP / 内网地址）
  ↓
按域名和路径分类内容类型
  ↓
分成两条支线：
  1. 音频/视频内容 → 下载到 inbox/
  2. 文字内容 → 直接写入 Obsidian Captures/
```

它不仅仅是“收链接”，而是先帮你做了预处理和内容识别。

#### Telegram 支持的内容分流

| 来源 | 处理方式 | 结果 |
|------|----------|------|
| YouTube / B 站 | 下载音频到 `inbox/` | 进入主处理管道 |
| TikTok / 抖音 | 下载音频到 `inbox/` | 进入主处理管道 |
| Instagram Reels | 下载音频到 `inbox/` | 进入主处理管道 |
| 小红书 / Twitter(X) | 下载音频到 `inbox/` | 进入主处理管道 |
| Threads | 抓取正文 | 直接写入 Captures |
| Instagram 图文帖 | 抓 caption | 直接写入 Captures |
| 通用网页 | 正文抽取 | 直接写入 Captures |

#### Telegram 这一层为什么厉害

- 用户只需要“把链接发出去”，系统就自动决定下一步
- 链接会被清洗，减少垃圾参数，后续引用更干净
- 不同平台有不同下载策略，不是单一粗暴抓取
- 会写 `.meta` sidecar，把来源 URL、平台、语言信息一起传给后续流程
- 处理完还能回发 Telegram 通知，形成闭环

`.meta` sidecar 的意义非常大，它让后面的 `process.py` 不只拿到一个文件，还知道：

- 这个文件来自哪个 URL
- 它原本属于哪个平台
- 用户希望按什么语言转录
- 是否需要走广告内容 prompt
- 是否要处理完成后通知指定 Telegram 用户

这让整个系统不再是“文件驱动”，而是“带上下文的任务驱动”。

### 入口 B：Plaud 自动拉取

文件：`pull-plaud.py`

这个入口解决的是“现实世界里大量会议和录音根本不是你手动导出来的”这个问题。

它通过 Plaud 网页接口自动从云端拉录音，不依赖你每次手动点下载。

完整流程：

```text
launchd 定时触发
  ↓
读取 ~/.plaud/config.json 中的 token
  ↓
访问 Plaud API 拉取录音列表
  ↓
过滤回收站内容和示例录音
  ↓
根据本地记录去重
  ↓
获取临时下载地址
  ↓
下载 MP3 到 inbox/
```

#### 这一层的价值

- 把 Plaud 从“设备”接成了“自动化数据源”
- 录完音后不用导出，不用手动同步，不用再打开 App
- 适合会议、访谈、远程通话、桌面录音等真实工作场景

---

### 入口 C：iCloud 录音收件箱

文件：`sync-icloud.py`

这个入口解决的是 iPhone 侧的语音捕获。

现实中最顺手的动作往往是：

1. 手机录音
2. 分享到文件
3. 保存进 iCloud 某个固定文件夹

系统会每 60 秒轮询一次 iCloud 指定目录，把新录音自动拉进 `inbox/`。

完整流程：

```text
launchd 每 60 秒执行
  ↓
扫描 iCloud Drive 里的录音收件箱
  ↓
跳过 .icloud 占位文件
  ↓
必要时强制下载云端文件
  ↓
等待文件大小稳定
  ↓
复制到本地 inbox/
```

#### 这一层为什么实用

- 让手机变成一个无摩擦采集端
- 不要求专门 App，也不要求特定录音软件
- “分享到文件”这个动作本身就足够自然

---

### 入口 D：手动拖入 / 浏览器直接落盘

这是最朴素但也最稳的入口。

所有其它自动化入口，本质上也都是在把内容送进 `inbox/`。所以即使某些来源暂时没接自动化，只要能把文件放到 `inbox/`，整个后续流程都仍然成立。

这代表系统的架构是松耦合的：

- 入口可以不断增加
- 核心处理管道不用改
- 只要统一投递到 `inbox/` 即可复用全部能力

## 第二阶段：统一进入主处理管道

文件：`process.py`

这是整套系统的大脑。

所有不同来源进来的内容，到了这里之后会被统一编排。

完整处理链如下：

```text
检测 inbox/ 新文件
  ↓
等待文件真正下载完成或改名完成
  ↓
读取同名 .meta sidecar（如果存在）
  ↓
移动到 processing/，避免重复处理
  ↓
如果是视频，先用 ffmpeg 提取音频
  ↓
用 ffprobe 获取时长
  ↓
按配置决定是否做 VAD 静音剥离
  ↓
调用 Vibe / sona CLI 做本地 Whisper 转录
  ↓
长音频自动切分处理
  ↓
根据来源和内容决定场景类型
  ↓
调用 Claude 生成标题
  ↓
调用对应 prompt 生成结构化总结
  ↓
写入 Obsidian Markdown
  ↓
保存 transcript 备份、记录状态、通知用户
```

---

## 第三阶段：本地转录

这里用的不是云转录，而是本地 Whisper 体系：

- `Vibe`
- `sona` CLI
- `whisper.cpp`
- CoreML / Metal 加速

这层设计的意义不只是“能转录”，而是兼顾了四个维度：

- 速度：Apple Silicon 本地跑，效率高
- 成本：不必每段音频都走云 API
- 隐私：录音不必上传第三方转录服务
- 稳定：网络波动不会卡死整条链路

### 转录阶段的工程细节

- 支持音频和视频多种格式
- 视频会自动提取出音轨
- 文件会等待到“大小稳定”再开始处理
- 如果下载器改了文件后缀，系统会追踪改名后的文件
- 可选 VAD 静音剥离，减少无效内容
- 长录音会分段转录，避免超长任务直接失败
- 可根据 `.meta` 或配置指定语言
- 支持 `auto` 自动检测语言

这一层说明系统考虑的不是“Demo 成功”，而是“真实世界文件到底有多脏、多乱、多不稳定”。

---

## 第四阶段：AI 理解内容，而不只是抄成文字

转录只是把音频变成文本，真正拉开差距的是后面的理解层。

### 1. 场景分类

系统不会把所有内容都用同一种模板处理，而是先判断内容属于哪一类。

已覆盖的场景包括：

- `meeting`
- `memo`
- `content`
- `1on1`
- `class`
- `call`
- `daylog`
- `discussion`
- `video`

如果是 Telegram 链接带来的音频内容，系统还能根据 `.meta` 直接走更贴合视频内容消费场景的 prompt。

### 2. 语义化标题生成

不是保留原始文件名，而是自动生成更像人写的标题。

这一步的价值非常高，因为长期知识库是否可读，很大程度取决于标题质量。

### 3. 按场景套用不同 prompt

不同内容类型会进入不同 prompt 模板。

| 场景 | Prompt | 输出重点 |
|------|--------|----------|
| `meeting` | `prompts/meeting.md` | 背景、讨论、决策、行动项 |
| `memo` | `prompts/memo.md` | 核心想法、延伸内容、下一步 |
| `content` | `prompts/content.md` | 核心观点、分主题拆解、结论 |
| `1on1` | `prompts/1on1.md` | 一对一对话重点 |
| `class` | `prompts/class.md` | 知识点和课堂结构 |
| `call` | `prompts/call.md` | 通话要点与后续动作 |
| `daylog` | `prompts/daylog.md` | 日志整理和事项收束 |
| `discussion` | `prompts/discussion.md` | 议题脉络与观点交锋 |
| `video` | `prompts/video-short.md` / `prompts/video-long.md` | 外部视频内容拆解 |

这说明系统不是“一个 prompt 打天下”，而是已经进入了内容工作流更成熟的阶段：**按任务类型进行结构化生产。**

---

## 第五阶段：写入 Obsidian，变成真正可积累的知识

最终输出不是一段控制台文字，而是写进 Obsidian 的 Markdown 文档。

每篇笔记都会包含这类信息：

- 日期
- 类型
- 来源文件
- 原始 URL
- 时长
- 标签
- AI 总结正文
- 完整转录原文

输出后的价值有三层：

- 能看：你当天就能快速回顾
- 能搜：之后可以按关键词、标签、日期检索
- 能复用：以后做总结、写作、复盘、汇报，都可以反复调用

这一步把“临时输入”变成了“长期资产”。

## 第六阶段：周报自动聚合

文件：`weekly-report.py`

这一步非常关键，因为它把单条笔记再提升一层，变成周期性的高层复盘。

完整流程：

```text
每周定时执行
  ↓
扫描过去 7 天的录音笔记
  ↓
跳过开发日志等非目标文件
  ↓
抽取每篇笔记的结构化正文
  ↓
剔除“完整转录”区，避免信息噪声过大
  ↓
把多篇笔记合并喂给 Claude
  ↓
生成一份本周周报
  ↓
写入 Obsidian 的 weekly/ 目录
```

这代表系统不只会“单次处理”，还会“周期性升维总结”。

也就是说，它做的不是单点自动化，而是连续性的知识运营。

## 支撑整套体验的辅助系统

### 1. 状态数据库

文件：`status_db.py`

系统使用 SQLite 状态库追踪每个任务的进度，并开启 WAL 模式，让写入和读取更顺滑。

可追踪的状态包括：

- `waiting`
- `extracting`
- `transcribing`
- `classifying`
- `titling`
- `summarizing`
- `saving`
- `done`
- `failed`

这层的意义是：它不再只是脚本，而开始具备“任务系统”的形态。

### 2. 菜单栏监控

文件：`menubar.py`

这个组件把后台流水线做成了更可感知的桌面体验：

- 常驻菜单栏
- 实时显示当前状态
- 展示今日完成数
- 展示最近处理记录
- 可直接打开 Obsidian 笔记

这会让系统从“看不见的自动化脚本”变成“你每天都能感受到存在感的工具”。

### 3. 失败兜底与防重复

真实可用的工作流，最重要的不是 happy path，而是边界情况。

这套系统已经考虑了不少工程细节：

- `processed.json` 做防重复记录
- 文件先进入 `processing/`，避免并发重复处理
- JSON 损坏时会备份并重建，不会因为单个文件损坏导致全线崩溃
- Claude 调用支持重试和超时控制
- 转录失败与总结失败可以拆开处理
- 原始 transcript 可单独保留，便于后续补救

这说明它不是“能跑一次”，而是“准备长期挂后台运行”。

## 这套系统真正厉害的地方

### 厉害点 1：它打通了“采集 → 理解 → 沉淀”全链路

很多工具只做其中一环：

- 有的只会录音
- 有的只会转录
- 有的只会总结
- 有的只会存笔记

这套系统把它们全部接成一条连续链路。

### 厉害点 2：它处理的是“真实工作流”，不是实验室输入

真实世界的内容来源非常杂：

- 手机
- 录音笔
- 桌面会议
- 社交媒体
- 网页内容
- 本地文件

而系统已经把这些脏乱复杂的来源统一到一个入口里。

### 厉害点 3：它默认后台运行，降低人的心智负担

最强的自动化，不是按钮多，而是你几乎感觉不到它，却持续帮你把事情做完。

这套系统借助 `launchd` 常驻运行，做到的是：

- 你不用记得“现在去处理”
- 你只要继续录、继续发、继续保存
- 其它事交给系统

### 厉害点 4：它不是“信息堆积”，而是“结构化产出”

同样是一段录音：

- 普通做法是得到一个文件
- 再好一点是得到一份转录
- 这套系统得到的是一篇带结构、带标题、带标签、带来源、可检索、可复盘的知识笔记

差别就在这里。

### 厉害点 5：它已经具备可扩展架构

因为系统采用“多入口统一投递到 inbox，再由核心管道统一处理”的结构，所以未来扩展会非常顺：

- 新增一个新的内容源，只需要做新的入口脚本
- 新增一个新的内容类型，只需要补一个新的 prompt
- 新增新的汇总层，只需要在 Obsidian 笔记基础上继续聚合

换句话说，这不是一段孤立脚本，而是一个可以继续长大的平台骨架。

### 版本 B：工作流视角
